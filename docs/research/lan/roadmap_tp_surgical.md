# TP-Surgical: Полный валидационный план

## 1. Что такое TP-Surgical и зачем

**Контекст.** Компании без датацентров, но с GPU-рабочими станциями по LAN,  
не могут запускать модели > 8B на одном устройстве. Покупать H100 за $30K+ —  
не вариант. Но 2–4 RTX 4090 по $1–1.5K уже есть у многих.

**Идея.** Разрезать трансформер по attention heads: каждая GPU — своя группа голов.  
Синхронизация через CCA-модули на «контрольных точках». Итог:  
один запрос обслуживается несколькими машинами в реальном времени по LAN.

## 2. Теоретическое обоснование скорости

### 2.1 Формула tok/s

Для одного токена декодирования при N GPU и K CCA sync-точках:

```
T_token = T_compute / N_gpu  +  K × RTT
tok/s   = 1000 / T_token  [мс]
```

где `T_compute` — время одного forward на одном GPU (мс), `RTT` — round-trip time.

### 2.2 Сходимость при RTT → 0

При RTT → 0:

```
T_token → T_compute / N_gpu
tok/s   → N_gpu × tok/s_single
```

**Вывод:** скорость теоретически сходится к линейному масштабированию.  
Ограничение — не алгоритм, а физика сети.

### 2.3 Break-even latency

Скорость split равна single GPU при:

```
RTT_breakeven = T_compute × (1 − 1/N_gpu) / K
```

Для Qwen3.5-0.8B (T_compute=36.6ms, N=2, K=7):

```
RTT_breakeven = 36.6 × (1 − 0.5) / 7 = 2.61 мс
```

Для Qwen3-8B (T_compute≈200ms, N=2, K=9):

```
RTT_breakeven = 200 × 0.5 / 9 = 11.1 мс
```

**Вывод:** чем больше модель, тем выгоднее split при одинаковой сети.  
На 8B даже 10ms RTT (дешёвый WiFi) уже окупается.

### 2.4 Measured numbers (RTX 4090, FLA installed)

| Сценарий | tok/s | vs single |
|---|---:|---:|
| Single GPU (baseline) | 34.1 | 1.0× |
| 2-GPU, RTT=0.01ms (NVLink) | 54.4 | +1.59× |
| 2-GPU, RTT=1ms (хороший LAN) | 39.5 | +1.16× |
| 2-GPU, RTT=2ms (наш LAN) | 30.9 | +0.91× |
| 2-GPU, RTT=5ms (медленный LAN) | 18.8 | -0.45× |

**Целевой RTT:** ≤ 1.5 мс одностороннего (3 мс RTT) на стандартном 10GbE свиче.

## 3. Scaling Laws для CCA

### 3.1 Аналогия с Chinchilla

Chinchilla (Hoffmann et al., 2022) показала: для LLM оптимально  
`N_tokens = 20 × N_params`. Иначе — либо underfitting (мало данных),  
либо compute-waste (слишком большая модель для данных).

Для CCA-дистилляции применяем ту же логику, но с поправкой:  
KL-distillation даёт **более богатый сигнал** чем next-token loss  
(soft distribution вместо 1-hot) → коэффициент ниже, примерно **5–10×**.

### 3.2 Формула CCA Scaling

```
N_tokens_optimal ≈ α × N_cca_params

где α ≈ 5–10  (distillation, soft labels)
    α ≈ 20    (Chinchilla, hard labels)
```

### 3.3 Расчёт для каждой модели

| Модель | H | CCA архитектура | CCA params | Нужно токенов | Нужно примеров (128 tok) |
|---|---:|---|---:|---:|---:|
| Qwen3.5-0.8B | 1024 | 7 × SwiGLU(H→H/2→H) | **5.5M** | 28–55M | 220–430K |
| Qwen3-4B | 2560 | 9 × SwiGLU(H→H/2→H) | **35M** | 175–350M | 1.4–2.7M |
| Qwen3-8B | 4096 | 9 × SwiGLU(H→H/2→H) | **90M** | 450–900M | 3.5–7M |

**Вывод:** для 0.8B нужно ~300K примеров OpenHermes. У нас пока 10K → фаза 1 является  
нижней оценкой качества. 100K (качаем сейчас) закрывает нижнюю границу.

### 3.4 Ожидаемая кривая качества

```
PPL recovery (%)
100% ┤                                         ●●●●
 90% ┤                               ●●●
 80% ┤                      ●●
 70% ┤             ●

      └──────────────────────────────────────────── токены (log scale)
      10K    100K    1M    10M    100M    1B
```

## 4. LAN-конфигурация для компании без датацентра

### 4.1 Минимальная конфигурация (2 узла)

Для Qwen3.5-0.8B или Qwen3-4B:

```
┌──────────────────┐         10GbE LAN         ┌──────────────────┐
│   Node 0 (GPU 0) │ ◄────────────────────────► │   Node 1 (GPU 1) │
│   RTX 4090 24GB  │         < 1ms RTT          │   RTX 4090 24GB  │
│   Worker: heads  │                            │   Worker: heads  │
│   [0 .. n/2)     │                            │   [n/2 .. n)     │
└──────────────────┘                            └──────────────────┘
         │                                               │
         └───────────────────┬───────────────────────────┘
                             │
                   ┌─────────▼──────────┐
                   │    Orchestrator    │
                   │  (может быть одним │
                   │   из Node 0/1)     │
                   │  - API endpoint    │
                   │  - KV-cache mgmt   │
                   │  - request routing │
                   └────────────────────┘
```

### 4.2 Что купить

| Компонент | Рекомендация | Цена (2026) |
|---|---|---:|
| GPU-узел × 2 | ПК с RTX 4090 24GB (или б/у 3090) | $2 000–3 000 × 2 |
| Сеть | 10GbE свич (TP-Link TL-SG108E или Mikrotik) | $50–100 |
| NIC | 10GbE карты в каждый ПК (Intel X550 или аналог) | $60–100 × 2 |
| Оркестратор | Отдельный мини-ПК (Intel NUC / Raspberry Pi 5) или Node 0 | $100–200 |
| Кабель | Cat 6A patch cable | $10 |

**Итого минимум: ~$4 500** (2 ПК с 4090 + сеть)  
**Для сравнения:** 1× H100 = $30 000+, аренда H100 = $2/ч

### 4.3 Почему 10GbE, а не обычный LAN

- Стандартный 1GbE: latency ~100–500 мкс, bandwidth 125 MB/s → **достаточно**  
  (данные CCA sync: 2KB per point, 7 points = 14KB/token — копейки по bandwidth)
- Реальная проблема — **latency**, не bandwidth
- 1GbE switch latency: 50–200 мкс = 0.05–0.2 мс → RTT ≈ 0.1–0.4 мс  
  → **обычный 1GbE офисный свич может быть достаточен!**
- 10GbE даёт стабильный sub-100мкс → RTT ≈ 0.1–0.2 мс (теоретически +2× vs single)

### 4.4 Оркестратор

Минимальный набор:
- gRPC endpoint для приёма запросов
- Распределение `input_ids` на оба Worker-а
- Сбор `logits` с Node 0 после последнего CCA
- KV-cache management (на каком узле хранится кэш)

Реализация: ~1–2 недели инженерии поверх текущего `playground/`.

## 5. Список гипотез для валидации

Каждая гипотеза — конкретный эксперимент с измеримым результатом.

### H1: CCA восстанавливает качество (главная)
> **Утверждение:** после KL-дистилляции PPL recovery ≥ 80% на held-out сете.

- **Эксперимент:** обучить CCA на 100K OpenHermes, измерить PPL до/после
- **Метрика:** `(ppl_split − ppl_cca) / (ppl_split − ppl_full) ≥ 0.8`
- **Срок:** 12 мая
- **Риск:** CCA переобучается → **митигация:** снизить CCA capacity, добавить данных

### H2: Scaling law выполняется
> **Утверждение:** PPL recovery растёт предсказуемо с ростом числа обучающих токенов.

- **Эксперимент:** 3 прогона: 10K, 50K, 200K примеров → построить кривую
- **Метрика:** кривая монотонна, аппроксимируется степенным законом
- **Срок:** 20 мая
- **Зачем:** позволяет предсказать бюджет данных для 4B и 8B без полного прогона

### H3: Реальный LAN RTT ≤ 1.5 мс
> **Утверждение:** офисный 10GbE свич даёт RTT ≤ 3 мс между двумя ПК.

- **Эксперимент:** `ping -i 0.001` между двумя узлами + `tc netem` для измерения baseline
- **Метрика:** `mean RTT ≤ 3.0 мс, p99 ≤ 5 мс`
- **Срок:** до первого деплоя
- **Риск:** офисный WiFi или managed switch дают 5–20 мс → нужен прямой провод

### H4: Скорость масштабируется с моделью
> **Утверждение:** для 4B и 8B моделей speedup выше, чем для 0.8B при одинаковом RTT.

- **Эксперимент:** playground benchmark для 4B и 8B с теми же network configs
- **Метрика:** speedup_4B > speedup_0.8B при RTT=2ms
- **Срок:** 25 мая (после H1 подтверждена)
- **Зачем:** 4090 × 2 для 8B даёт бо́льший выигрыш → лучший ROI для компании

### H5: CCA веса переносятся между fine-tuned версиями
> **Утверждение:** CCA, обученный на base модели, работает на SFT-версии без переобучения.

- **Эксперимент:** применить base-CCA к Qwen3.5-0.8B-Instruct, измерить PPL recovery
- **Метрика:** recovery ≥ 50% без переобучения (≥ 80% после 1000 шагов дообучения)
- **Срок:** после H1
- **Зачем:** один CCA на семейство моделей → значительно снижает стоимость

### H6: Минимальное число CCA точек
> **Утверждение:** 4 CCA точки дают ≥ 90% от качества 7 CCA точек при RTT=2ms.

- **Эксперимент:** обучить CCA с [3,7,15,23] vs [0,3,7,11,15,19,23], сравнить PPL
- **Метрика:** `ppl(K=4) / ppl(K=7) ≤ 1.10`  
- **Зачем:** меньше RTT → выше скорость; если quality держится — K=4 предпочтительнее

## 6. Конкретный план валидации

### Неделя 1 (5–12 мая): 0.8B quality baseline

```
День 1–2:  Training run на RTX 4090 (10K примеров, 60K шагов)
           → первая оценка PPL recovery
День 3–4:  Если recovery > 0%: scaling run с 50K примерами
           Если recovery ≤ 0%: ablation CCA capacity / LR / schedule
День 5–6:  Playground benchmark с реальным `tc netem` latency
Day 7:     Go/No-Go: зафиксировать H1, написать результаты
```

**Вычисления:**
- RTX 4090: 10K примеров → 60K steps → ~2ч
- RTX 4090: 50K примеров → 60K steps → ~2ч (данные на токенизацию быстро)
- Если нужна большая скорость итерации: H100 1ч = $1.50

### Неделя 2–3 (13–25 мая): Scaling curve + 4B

```
День 8–10:  Scaling run: 10K / 50K / 200K примеров → кривая H2
День 11–14: Activation profiling Qwen3-4B
            Training CCA 4B (H100, ~3ч, $4.50)
            Playground benchmark 4B
День 15:    Сравнение: 0.8B vs 4B recovery и speedup
```

### Неделя 4 (26–30 мая): 8B + деплой-прототип

```
День 16–18: Training CCA 8B (H100, ~6ч, $9)
            Проверка H4: speedup 8B vs 0.8B
День 19–20: Минимальный distributed runtime (2 воркера + координатор)
            Реальный latency тест между двумя машинами
День 21:    Демо: end-to-end запрос через 2 узла
```
