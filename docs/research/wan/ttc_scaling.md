# Time-Traveling Columns (TTC): Масштабирование по блокам

**Цель:** Разработать архитектуру, где P2P-сеть предсказывает токены "вперед" (T+1, T+2...), чтобы спрятать пинг за параллельным батчингом.

## 1. Концепция TTC (Time-Traveling Columns)

В классической авторегрессии модель предсказывает только следующий токен ($T+1$). В архитектуре Medusa/EAGLE к модели добавляются дополнительные "головы" (heads) или блоки, которые пытаются предсказать $T+2, T+3$ и т.д., опираясь на текущий скрытый вектор (hidden state).

В нашей децентрализованной версии (TTC) мы выносим эти предсказывающие блоки на удаленные воркеры.

![TTC Architecture](../diagrams/ttc_architecture.png)

### Как это работает:
1. **Local Stem (Локальный ствол):** На устройстве пользователя крутится базовая модель (например, Qwen 0.5B). Она генерирует скрытый вектор контекста (Context Vector) и предсказывает токен $T+1$.
2. **Stateless Workers (Удаленные воркеры):** Локальная нода рассылает вектор контекста в P2P-сеть.
   - Воркер 1 (Колонка 1) получает вектор и предсказывает токен $T+2$.
   - Воркер 2 (Колонка 2) получает вектор и предсказывает токен $T+3$.
   - Воркер 3 (Колонка 3) получает вектор и предсказывает токен $T+4$.
3. **Aggregator (Сборщик):** Локальная нода получает ответы от воркеров (за один пинг) и собирает из них дерево возможных продолжений. Если предсказания верны, мы принимаем сразу 3-4 токена за один сетевой раунд-трип.

## 2. Масштабирование: Как выглядят TTC блоки для разных размеров (xB)

Главная прелесть TTC в том, что удаленные воркеры **не обязаны быть полными копиями большой модели**. Они могут быть маленькими, специализированными блоками. Мы можем оперировать блоками по **~100 млн параметров**.

### Вариант А: Легкая сеть (Local 0.5B + Remote 100M blocks)
*Сценарий: Пользователь запускает модель на ноутбуке, сеть состоит из слабых GPU.*

- **Local Stem:** Qwen 0.5B (занимает ~1 ГБ VRAM).
- **Remote TTC Blocks:** Каждая колонка — это маленькая MLP-сеть или 1-2 слоя трансформера поверх замороженного эмбеддинга. Размер: **100M параметров** (~200 МБ VRAM).
- **Топология:** 
  - Worker 1: 100M block (T+1)
  - Worker 2: 100M block (T+2)
  - Worker 3: 100M block (T+3)
- **Плюсы:** К сети могут подключаться даже старые видеокарты (GTX 1060). Передача весов воркерам занимает секунды.

### Вариант Б: Средняя сеть (Local 1.5B + Remote 300M blocks)
*Сценарий: Пользователь с хорошим ПК, сеть из средних GPU (RTX 3060).*

- **Local Stem:** Qwen 1.5B (занимает ~3 ГБ VRAM).
- **Remote TTC Blocks:** Каждая колонка — это 4 слоя трансформера (Layer Slice). Размер: **~300M параметров** (~600 МБ VRAM).
- **Топология:**
  - Worker 1: 300M block (T+1)
  - Worker 2: 300M block (T+2)
  - Worker 3: 300M block (T+3)
- **Плюсы:** Более глубокие блоки могут предсказывать более сложные логические связи на 2-3 шага вперед, повышая Acceptance Rate.

### Вариант В: Тяжелая сеть (Local 0.5B + Remote 7B Verifiers)
*Сценарий: Пользователь на телефоне/слабом ПК, сеть из мощных майнинг-ферм (RTX 3090/4090).*

В этом случае мы переворачиваем архитектуру. Локальная нода слишком слаба, чтобы быть "мастером" для большой модели. Мы используем классический **Distributed Speculative Decoding**.

- **Local Stem:** Qwen 0.5B (генерирует черновик из 5 токенов).
- **Remote TTC Blocks:** Полноценная Qwen 7B (занимает ~14 ГБ VRAM).
- **Топология:** Локальная нода кидает черновик на удаленный 7B-воркер. Воркер делает один forward pass и возвращает проверенные токены.
- **Плюсы:** Пользователь со слабым железом получает качество 7B модели со скоростью 10-15 tok/s.

## 3. Анатомия TTC-блока (100M параметров)

Как выглядит идеальный удаленный воркер на 100 млн параметров? Он не должен быть полноценной LLM. Он должен быть "предсказателем будущего" на основе чужого контекста.

**Входные данные (Payload по сети):**
- Скрытый вектор последнего слоя Local Stem (размерность `d_model`, например 1024).
- Токен, который мы сейчас проверяем.

**Архитектура блока (TTC Block):**
```text
Input Vector (1024)
→ RMSNorm
→ 1x Cross-Attention (Query = Input Vector, Key/Value = Local KV-cache projection)
→ 1x SwiGLU MLP (Hidden = 4096)
→ RMSNorm
→ Linear (Vocab Size = 151,936)
→ Output Logits
```

**Почему 100M?**
- Матрица словаря (1024 * 151,936) = ~155M параметров (если не использовать Tie Embeddings).
- Внутренние слои (Attention + MLP) = ~10-20M параметров.
- Итого: очень легкая "голова", которая быстро скачивается и мгновенно работает даже на слабом железе.

## 4. Главный инженерный вызов для TTC

Чтобы TTC-блоки работали, им нужен **контекст**. Если локальная нода будет пересылать весь KV-кэш (историю чата) по сети на каждый запрос, мы снова умрем от Bandwidth Death.

**Решение: Prompt Compression / Context Vectoring**
Локальная нода не шлет весь KV-кэш. Она шлет только **один сжатый вектор** (Hidden State последнего токена) или небольшую выборку (Top-K attention scores). TTC-блок обучается (через дистилляцию) предсказывать будущее, опираясь *только* на этот сжатый вектор.
